Section 1

Clean Code Principles

Why clean code matters

Code is read far more often than it's written. Every hour spent writing clear code saves ten hours of debugging and maintenance later.

The goal: Code should read like well-written prose. If you need comments to explain what it does, the code itself isn't clear enough.

Naming conventions

  • Variables: Nouns that describe what they hold — userCount, maxRetries
  • Functions: Verbs that describe what they do — getUserById(), calculateTotal()
  • Booleans: Predicates — isValid, hasPermission, canEdit
  • Classes: Nouns — UserRepository, PaymentProcessor

Function guidelines

FUNCTIONS SHOULD: • Be small (< 20 lines ideal) • Do one thing and do it well • Have few arguments (0-2 ideal, 3 max) • Follow command-query separation • Have no side effects (when possible)
PythonGood vs Bad naming
# Bad: What does this do?
def proc(d, f):
    return [x for x in d if f(x)]

# Good: Crystal clear intent
def filter_active_users(users, is_active):
    return [user for user in users if is_active(user)]

SOLID Principles

Five principles for maintainable object-oriented design:

S — Single Responsibility: One class, one reason to change O — Open/Closed: Open for extension, closed for modification L — Liskov Substitution: Subtypes must be substitutable I — Interface Segregation: Many specific interfaces > one general D — Dependency Inversion: Depend on abstractions, not concretions
Common violation: A User class that handles authentication, database access, AND email sending. Split these into Authenticator, UserRepository, and EmailService.
PythonDependency Inversion example
# Bad: Concrete dependency
class OrderService:
    def __init__(self):
        self.db = MySQLDatabase()  # Locked to MySQL!

# Good: Depend on abstraction
class OrderService:
    def __init__(self, database: Database):
        self.db = database  # Any database works!
Section 3

Testing Strategies

The testing pyramid

/\ E2E Tests (few, slow, expensive) / \ /----\ Integration Tests (some) / \ /--------\ Unit Tests (many, fast, cheap)

Unit tests form the foundation. They're fast, isolated, and catch bugs early. Integration tests verify components work together. E2E tests validate complete user flows.

The AAA pattern

Arrange → Act → Assert: Set up the test, perform the action, verify the result.
PythonAAA pattern in action
def test_add_item_to_cart():
    # Arrange
    cart = ShoppingCart()
    item = Product("Widget", price=9.99)
    
    # Act
    cart.add(item)
    
    # Assert
    assert len(cart.items) == 1
    assert cart.total == 9.99

What to test

  • Happy path: Normal expected usage
  • Edge cases: Empty inputs, nulls, maximum values
  • Error conditions: Invalid inputs, network failures
  • Boundary conditions: Off-by-one, limits
Section 4

Systematic Debugging

The scientific method

1. OBSERVE — Reproduce the bug reliably 2. HYPOTHESIZE — Form a theory about the cause 3. PREDICT — What should happen if hypothesis is true? 4. EXPERIMENT — Test your prediction 5. ANALYZE — Refine hypothesis and repeat

Debugging techniques

  • Binary search: Halve the problem space each step
  • Rubber duck: Explain the code line by line out loud
  • Print debugging: Strategic logging to trace execution
  • Breakpoints: Pause execution and inspect state
  • Git bisect: Find the commit that introduced the bug
Pro tip: When you find a bug, write a test that reproduces it BEFORE you fix it. This prevents regression and proves the fix works.

Common bug patterns

  • Off-by-one: Loop bounds, array indices
  • Null/undefined access: Missing null checks
  • Mutation bugs: Unexpected state changes
  • Race conditions: Timing-dependent failures
Section 5

Performance Optimization

Knuth's law: "Premature optimization is the root of all evil." First make it work, then make it right, then make it fast.

Big-O reference

O(1) — HashMap lookup, array access O(log n) — Binary search, balanced tree ops O(n) — Linear scan, single loop O(n log n) — Merge sort, heap sort O(n²) — Nested loops, bubble sort O(2ⁿ) — Subsets, naive recursion O(n!) — Permutations, brute force

Optimization strategies

  • Better algorithm: Always beats micro-optimization
  • Caching: Trade memory for speed
  • Lazy evaluation: Don't compute until needed
  • Batching: Reduce overhead with bulk operations
  • Right data structure: HashSet for lookups, not List
Measure first! Always profile before optimizing. Your intuition about bottlenecks is often wrong.
