Reductions and Hardness

Prove problems are hard by showing they're at least as hard as known hard problems.

01

Reductions

Reduction: Problem A reduces to B if solving B lets us solve A.
A ‚â§ B means: "A is no harder than B" "If we can solve B, we can solve A" "If A is hard, then B is at least as hard" POLYNOMIAL-TIME REDUCTION (‚â§‚Çö): Transform A's input to B's input in poly-time Use B's answer to get A's answer

Direction matters! A ‚â§ B proves B is hard (if A is hard), not that A is hard.

Complexity Classes

P (Polynomial Time): Problems solvable in O(n^k) time Example: Sorting, shortest path, primality NP (Nondeterministic Polynomial): Problems verifiable in polynomial time Given a "certificate", can check if correct quickly Example: SAT, Hamiltonian Path, Subset Sum P ‚äÜ NP (obvious) P = NP? (the million-dollar question)
03

NP-Completeness

NP-HARD: At least as hard as every problem in NP Every NP problem reduces to it NP-COMPLETE: NP-hard AND in NP The "hardest" problems in NP IF any NP-complete problem has poly-time solution, THEN P = NP (all NP-complete problems become easy)
Cook-Levin Theorem: SAT is NP-complete. (The first one proven!)

Coping with NP-Hardness

IF PROBLEM IS NP-HARD: 1. Approximation algorithms (get close to optimal) 2. Heuristics (work well in practice) 3. Parameterized complexity (fast if parameter is small) 4. Special cases (maybe P on restricted inputs) 5. Randomization (Monte Carlo algorithms) 6. Exponential but practical (branch and bound)
